# Verifying Your Setup

This guide walks you through validating that your **generated GORM CLI APIs and field helpers** work correctly in your project. By following these steps, you will ensure your code generation was successful and that the type-safe query interfaces and model-driven helpers produce expected results with real data.

---

## 1. Confirm Generated Code Exists

After running the `gorm gen` command (see [Generating Code](/getting_started/first_code_generation/generating_code)), start by verifying your output directory (default `./g` or your configured path) contains generated Go files.

- Look for files with code for interfaces (e.g., `query.go`) and model fields (e.g., `user.go`).
- Confirm these files are **not empty** and contain definitions like `type Query[T any]` or variables such as `User` representing model fields.

<Tip>
Missing or empty files usually indicate a generation or config issue. Review your input files and `genconfig.Config` settings if generation failed.
</Tip>

## 2. Integrate the Generated Code

Import the generated package in your Go project as you would any local code:

```go
import (
  "context"
  "your_project/generated"
  "gorm.io/gorm"
  "your_project/models"
)
```

<Tip>
Ensure your Go module structure references the correct path to generated code and modules, and that your `go.mod` is updated.
</Tip>

## 3. Instantiate the Query Interface

Create an instance of your generated query interface bound to your GORM database:

```go
func runFirstQueries(db *gorm.DB) error {
  ctx := context.Background()

  // Instantiate the Query interface for the User model
  userQuery := generated.Query[models.User](db)

  // Execute simple queries below...

  return nil
}
```

This sets you up to use the type-safe methods with your database.

## 4. Run Sample Queries

Try these example queries to confirm your API behaves as expected:

### a. Fetch by ID

```go
user, err := userQuery.GetByID(ctx, 123)
if err != nil {
  return err
}
fmt.Printf("User ID 123: %+v\n", user)
```

Expect a valid `User` struct or an error if not found.

### b. Filter by Column

```go
result, err := userQuery.FilterWithColumn("name", "alice")
if err != nil {
  return err
}
fmt.Printf("User named alice: %+v\n", result)
```

The dynamic column query should return a matching user.

### c. Conditional Query

```go
userFilter := models.User{Name: "bob"}
userResult, err := userQuery.QueryWith(ctx, userFilter)
if err != nil {
  return err
}
fmt.Printf("Query with conditional where: %+v\n", userResult)
```

This uses SQL templates with conditions mapped to struct fields.

### d. Updating Records

```go
userUpdate := models.User{Name: "updated_name", Age: 30}
err = userQuery.UpdateInfo(ctx, userUpdate, 123)
if err != nil {
  return err
}
fmt.Println("User updated successfully")
```

Uses the generated update method with dynamic `SET` clauses.

### e. Filtering Collections

```go
users := []models.User{
  {Name: "alice", Age: 25},
  {Name: "charlie", Age: 35},
}
results, err := userQuery.Filter(ctx, users)
if err != nil {
  return err
}
fmt.Printf("Filtered users: %+v\n", results)
```

Tests iteration and multiple conditions in SQL templates.

## 5. Interpret Output

- Successful query methods return typed model instances or slices alongside `error`.
- For update methods, an `error` result indicates success or failure.

Ensure no panics or unexpected empty results occur.

<Tip>
Use `fmt.Printf("%+v\n", result)` during early tests to inspect outputs and validate correctness.
</Tip>

## 6. Debugging Common Issues

### a. Method Not Found or Compilation Errors

- Confirm the generated code is imported correctly and build is clean.
- Check your SQL templates in interfaces for syntax errors or unsupported constructs.

### b. Query Returns Empty Results

- Verify your database has test data matching query criteria.
- Make sure your model struct tags map to correct table and column names.

### c. Update Operations Not Applying

- Examine generated SQL with debugging or `db = db.Debug()` in GORM.
- Check that conditional set clauses are triggered by non-empty fields.

### d. Configuration-Related Filtering

- If some interfaces or models are not generated, review your `genconfig.Config` for includes/excludes.

<Tip>
Use `scripts/test_all.sh` (if available) to run automatic tests validating your configuration and generation correctness.
</Tip>

## 7. Example: Running Integration with Your Database

```go
package main

import (
  "context"
  "fmt"
  "gorm.io/driver/mysql"
  "gorm.io/gorm"
  "your_project/generated"
  "your_project/models"
)

func main() {
  dsn := "gorm:gorm@tcp(127.0.0.1:3306)/gorm?parseTime=true&charset=utf8mb4&loc=Local"
  db, err := gorm.Open(mysql.Open(dsn), &gorm.Config{})
  if err != nil {
    panic(err)
  }

  ctx := context.Background()
  q := generated.Query[models.User](db)

  user, err := q.GetByID(ctx, 1)
  if err != nil {
    fmt.Println("Error fetching user:", err)
    return
  }

  fmt.Println("Fetched user:", user)
}
```

Replace DSN and package paths with your configuration to run successful queries.

## Troubleshooting Summary

| Issue                          | Diagnosis                                               | Solution                                                      |
| ------------------------------| ------------------------------------------------------ | -------------------------------------------------------------|
| Generated code missing files   | Input path, config excludes, or CLI errors             | Verify input path and config; rerun `gorm gen` with logs      |
| Compilation errors after gen   | Incorrect imports or broken interface definitions       | Check import paths; review SQL comments for syntax           |
| Queries return no data         | No matching rows or mapping issues                       | Confirm DB contents, verify model tags and table mappings    |
| Update calls do nothing        | Conditional `set` clauses not triggered or DB permissions| Confirm data triggers conditions; check user DB write access |

<Tip>
Run your existing unit or integration tests against generated code, or use provided `scripts/test_all.sh` to execute module tests as an automated validation step.
</Tip>
